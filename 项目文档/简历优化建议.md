# 轻食AI项目简历优化建议（面试导向）

## 1. 你当前版本的优点
- 覆盖了完整技术栈，能体现你有全链路能力。
- 功能描述较全，已经提到登录、识图、建议、聊天等核心功能。
- 有“工程化表达”意识（如拦截器、RESTful、状态持久化）。

## 2. 主要问题（为什么面试官会追问）
- **问题1：业务价值不够聚焦**  
  当前写法“功能罗列”偏多，缺少“这个项目解决了什么问题”的一句话价值。
- **问题2：量化不足**  
  大多是“做了什么”，较少“做完后效果如何”，面试官很难判断产出规模。
- **问题3：技术名词堆叠**  
  例如 AI-SDK/OpenAI SDK/视觉模型等并列出现，但没有说明分工关系，容易被问穿。
- **问题4：部分措辞风险偏高**  
  比如“自动入库/同步”如果没有补充校验策略，会被追问数据一致性与异常处理。

## 3. 建议的简历改写版本（可直接替换）

## 轻食AI｜基于大模型的健康饮食助手（前后端开发负责人） 2026.01 - 2026.03
- 面向个人用户设计健康饮食管理平台，打通“餐食记录（手填/识图）→ 数据入库 → 仪表盘展示 → AI 个性化建议/对话咨询”的业务闭环。
- 基于 React + TypeScript + Ant Design + Zustand 构建前端，完成登录注册、仪表盘、饮食分析、对话咨询、个人中心等核心模块；通过 Zustand + LocalStorage 实现登录态与当日建议持久化，减少重复请求。
- 封装 Axios 请求层与拦截器，统一 token 注入与异常处理；按业务域拆分 API（登录、饮食、分析、聊天、个人中心），提升接口调用一致性与可维护性。
- 基于 Express + Sequelize + MySQL 设计并实现后端服务，覆盖用户、健康档案、饮食记录、聊天记录等核心模型；通过业务校验 + 联合唯一约束保障同用户同日同餐次数据一致性。
- 接入多模型能力：视觉模型用于餐食识别与热量估算、对话模型用于健康问答与流式输出、建议模型用于结构化饮食建议生成，实现 AI 能力在真实业务中的落地。
- 支持对话工具调用读取当日饮食记录，使 AI 建议从通用回答升级为“基于用户真实数据”的个性化回答。

## 5. 逐条修改理由（你可以这样向面试官解释）
- “项目介绍”改为“问题 + 目标 + 闭环”，让面试官先理解业务，再问技术细节。
- 把“技术栈”融入“做了什么”里，避免孤立堆名词，增强可读性。
- 增加“策略词”：统一、闭环、一致性、结构化、个性化，体现工程化思维。
- 把“AI 接入”拆成“识图/对话/建议”三段，便于回答“为什么这么选型”。
- 强化“数据一致性”描述，提前回应后端面试官最关心的可靠性问题。

## 6. 面试追问准备（按你这版简历最可能被问）
- 为什么选择 Zustand，而不是 Redux Toolkit？
- 识图结果不稳定时，你怎么做校验和兜底？
- 聊天工具调用是怎么触发的？怎么避免每条消息都查数据库？
- 联合唯一索引与业务层去重分别解决什么问题？
- 你在这个项目里最有挑战的一次故障/bug 是什么，如何定位并修复？

## 7. 风险措辞替换建议
- 将“自动入库/同步”改为“经校验后入库并在仪表盘展示”，避免绝对化承诺。
- 将“实现闭环”补充为“完成核心链路闭环”，避免被问到边界功能。
- 将“统一异常处理”补充为“前端统一处理常见异常码”，避免被追问到全链路监控。
