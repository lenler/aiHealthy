#### 请求拦截器

1. 统一 token 注入,全局异常处理

再说拦截器, 我在http实例中配置了请求拦截器和响应拦截器,在这两个拦截器中做了不同的处理,在请求拦截器 interceptors.request.use中, 我主要做了token信息的注入, 每次请求都会从本地获取 token信息并写入请求头`[Authorization]`中统一注入token.  并且在拦截器中添加一个异常处理器 处理请求异常, 例如网络异常等, 通过全局异常处理器进行统一处理, 例如弹出提示框等

```ts utils/http
http.interceptors.request.use((config)=>{
    //为请求头添加token
    const token=localStorage.getItem('token')
    //如果token不存在 则直接返回config 拒绝请求
    if(!token){
        return config
    }
    //并不是所有页面都需要token
    if(token){
        //http协议中的标准字段 专门用来携带认证信息
        config.headers['Authorization']=`${token}`
    }
    return config
}, (error) => {
    return Promise.reject(error);
})
```

1. post请求去重

完整流程:

- 第一步，请求进入拦截器 ：判断是不是 post
- 第二步，生成请求标识 ：例如 post-/api/analysis/upload
- 第三步，检查 Map
  - 如果已存在：直接拦截，返回“请勿重复提交请求”
  - 如果不存在：写入 Map，继续发送请求
- 第四步，请求结束后释放锁
  - 成功响应时删除 key，在 response interceptor success
  - 请求失败时也删除 key，在 response interceptor error
- 这样就能保证： 同一个 POST 请求在完成前，不能再次提交

我对 POST 请求做了请求层去重。具体做法是在 Axios 拦截器里维护一个 pendingRequests Map，用请求方法和 URL 生成唯一 key。请求发送前先检查这个 key 是否已存在，如果存在就直接拦截，避免重复请求打到后端；如果不存在，就先记录下来，等响应成功或失败后再删除。这样可以在请求层解决连点、弱网重发和脚本重复触发的问题，比单纯按钮禁用更稳

### 后端get接口缓存优化

后端只读接口缓存头设置
实现的缓存机制:

- 强缓存 ：中间件统一返回 Cache-Control: private, max-age=秒数 和 Expires
- 鉴权隔离 ：返回 Vary: Authorization ，避免不同登录态共享同一份缓存
- 协商缓存 ：重写 res.json ，对响应体做 JSON.stringify ，再用 sha1 生成 ETag
- 304 返回 ：如果请求头里的 If-None-Match 和当前 ETag 一致，服务端直接返回 304，不再下发响应体

达到的效果:

- 短时间重复访问更快 ：dashboard/person 的读接口在缓存周期内可直接命中浏览器缓存
- 减少重复流量 ：缓存过期后，浏览器会带上 If-None-Match ，数据没变时服务端只回 304
- 降低服务端压力 ：减少重复 JSON 传输与部分重复渲染链路开销
- 保证数据边界 ：只对读多写少接口启用，analysis 这类实时性强的接口不缓存

我在后端,通过一个工具函数,将一些关于强缓存和协商缓存的请求头配置封装,为get接口做了两层缓存优化。第一层是短时强缓存，对 dashboard(仪表盘展示) 的 GET 设置 30 秒，对 person(个人账户信息) 的 GET 设置 60 秒，通过 Cache-Control 和 Expires 让浏览器在短时间内直接复用本地缓存。第二层是协商缓存，我在服务端对 JSON 响应计算 ETag，并在下次请求时校验 If-None-Match 。如果数据没变，服务端直接返回 304，不再传输响应体。这样既降低了高频访问压力，也兼顾了数据更新后的正确性。

## 简历第二条

基于 Zustand + LocalStorage 构建登录态与 AI 建议缓存机制，支持刷新后状态恢复与按日期复用建议结果，减少重复请求并提升页面访问连续性

我用 Zustand 做了两类全局状态管理：一类是登录态，一类是 AI 建议缓存。登录态这块，我把 token、昵称和 userId 放进 authStore，并同步持久化到 localStorage，这样用户刷新后不用重新登录，路由守卫也能直接基于 token 做访问控制。建议缓存这块，我在 infoStore 里持久化了 aiAdvice、targetCalories 和 lastFetchDate，并在首页加载时先判断今天是否已经拉取过建议，如果已经拉过就直接用缓存，没有才重新请求，所以减少了重复请求，也保证了刷新后的连续体验。

有价值的点:

- 减少重复请求 ：当天 AI 建议不会每次进入首页都重新请求
- 刷新后状态可恢复 ：用户刷新页面后，登录态和建议数据还能直接使用
- 提升体验连续性 ：避免刷新后重新登录、重新拉建议带来的闪烁和等待
- 统一状态入口 ：把用户信息和建议数据从页面局部状态提升到全局 store，减少状态分散

### 登录管理

我在Zustand是按照业务划分store的 所以我在Zustand中划分了一个authStore 用来维护token,用户昵称,用户id等常用信息,并在初始化的时候将localstore中的信息回填入store中.

- 封装了 login / logout / updateUserInfo 三个动作：
  - login 时把 token、昵称、userId 同步写入 Zustand 和 localStorage
  - logout 时统一清空本地存储和全局状态
  - 更新昵称时同步更新 store 和本地缓存
  - 在登录页 login/index.tsx 登录成功后，调用 store 的 login 方法完成全局登录态写入
  - 在路由守卫 routerDefence/index.tsx 里读取 Zustand 中的 token，统一做未登录跳转，避免每个页面重复写鉴权逻辑

### 建议缓存

在 infoStore.ts里 我使用了Zustand的持久化中间件 persist 吧aiAdvice 目标卡路里,建议更新日期都持久化到localStorage中,
这样做不是单纯为了把数据存起来,而是加了一个按照日期控制是否重新拉去建议的策略: 我的做法是在仪表盘(首页),当用户第一次进入页面的时候就会比较,lastFetchDate和当前的日期,如果是同一天就直接是同store中的缓存,如果是今日 但是已经存在缓存数据了 就直接用store中的数据.
请求成功后，再通过 setAiData 和 updateLastFetchDate 回写缓存，形成“读取缓存 → 判断是否过期 → 必要时刷新”的闭环

## 简历第三条

设计 AI 识图上传与结果确认链路，引入上传状态锁、草稿暂存与人工二次确认机制，减少误提交与无效入库，提升识别交互的稳定性与用户操作可控性。

这个页面的问题不是把图片传上去，而是怎么避免 AI 识别过程中的误操作。因为如果识别完成后直接写库，用户一旦上传错图、识别错餐次或者连续点击，就会把脏数据落进正式记录。所以我把这个流程拆成了两阶段：第一阶段是上传和识别，结果先进入草稿区；第二阶段是人工确认，只有用户显式点击确认后才会真正调用新增或更新接口。与此同时，我还加了上传中的状态锁和页面离开拦截，避免重复上传和中途中断。这样就把高风险操作从‘自动入库’改成了‘先识别、再确认’的受控流程

### 工作链路

我设置了一个状态用来确认上传状态, 当用户点击上传后,setIsRqeuire(true) 同时给上传组件设置disabled属性,禁止用户再次点击上传按钮,当上传完成后,无论成功还是失败,都将isRequire设置为false 解除上传锁. 这样就能有效避免用户在上传过程中重复点击导致的多次请求和数据混乱问题。
然后我将Ai返回的结果落成草稿数据 并默认全选 只有当用户确认添加到餐次后才会真正提交到后端, 这样就能有效避免误提交和无效入库的问题, 同时也提升了用户对识别结果的可控性和交互的稳定性.
同时我还做了一些兜底操作:

1. 用户没有选择任何结果点击确认,系统会提示用户至少选择一个结果,避免无效提交
2. 支持一键清空结果,方便用户重新选择,提升交互体验
3. 上传过程中监听beforeunload时间,防止用户刷新或关闭页面 导致任务中断, 提示用户正在上传中,是否要放弃上传,提升交互的稳定性和用户体验

```tsx
  // 添加页面关闭/刷新拦截
  useEffect(() => {
    const handleBeforeUnload = (e: BeforeUnloadEvent) => {
      if (isRequire) {
        e.preventDefault();
        e.returnValue = ''; // Chrome 需要设置 returnValue
      }
    };

    window.addEventListener('beforeunload', handleBeforeUnload);
    return () => {
      window.removeEventListener('beforeunload', handleBeforeUnload);
    };
  }, [isRequire]);

```

## 简历第四条

按识图、问答、建议三类场景接入**视觉模型**、**对话模型**与**结构化生成模型**，并通过 **Prompt 约束**、**JSON 输出控制**与**后端兜底解析**提升大模型输出稳定性，支撑识别结果展示与建议生成。

我在项目里没有把大模型当成一个统一黑盒，而是按场景拆成了三类：识图场景接视觉模型，问答场景接对话模型，建议场景接结构化生成模型。因为不同场景对模型能力要求不一样，比如识图需要多模态能力，问答更强调流式体验，建议生成则更强调结构化输出稳定性。为了保证模型结果能直接进入业务链路，我做了三层控制：第一层是 Prompt 约束，让模型只能返回指定 JSON；第二层是接口层通过 response_format 强化结构化输出；第三层是后端解析和默认值兜底，保证即使模型输出异常，前端依然拿到可渲染、可入库的数据。

### 工作链路

我吧AI能力分成了三个模型,各司其职.

- 一个用来负责视图场景的视觉模型,用来识别食物类型和热量估算
- 一个用来负责对话场景的对话模型,做健康问答的流式输出
- 一共用来在建议场景的结构化生成建议的模型, 返回饮食建议和建议目标热量

在建议模型和视图模型我都做了大模型的输出控制,首先是设置一个合理的prompt明确输出格式与字段,但是这样做是很不保险的 所以在我研究模型的调用接口之后,我在模型配置中添加了一个`respose_format:{type: "json_object"}`强化结构化输出.
这个字段的作用是强制ai返回一个JSON结构的对象

即使这样ai又是也会返回一个不规范的结果 所以我处理在模型调用的时候做出限制之外,还添加了一个后端兜底的逻辑, 我会视图结果先解析 然后通过Array.isArray判断, 字段整理,和默认值兜底的防止保证视图模型的输出是可靠的. 在建议模型中 我做了一个兜底逻辑 如果JSON解析失败或是结果不符合预期,我会返回一个默认的建议结果或是预存在缓存中的数据, 这样就能保证即使模型输出不稳定,用户也能拿到一个合理的建议,提升了整体交互的稳定性和用户体验.

## 简历第五条

基于 **AI SDK** 与 **DeepSeek** 实现**健康问答流式输出**，并通过**工具调用**读取用户当日饮食记录，使 AI 回答能够结合真实业务数据生成个性化饮食建议。

我在问答模块里而是基于 AI SDK 和 DeepSeek 实现了流式对话能力。用户发送问题后，后端通过 streamText 把模型输出实时推给前端，前端用 AI SDK提供的 useChat 配合ANTDX提供的气泡框做流式渲染，保证交互上有打字机效果。我没有让模型只基于通用语料回答，而是给它接了一个工具调用能力：当用户询问今日饮食情况时，模型会先调用后端工具去查询数据库里的当日饮食记录，再结合这些真实数据生成个性化建议。这样就把 AI 从通用问答升级成了和业务数据打通的智能营养顾问

### 工作链路

在后端我没有使用自己搭的sse来做消息推送, 而是使用了AI SDK提供的streamText方法, 这个方法是专门用来做流式输出的, 通过pipeUIMessageStreamToResponse这个工具函数将模型返回的消息提供给前端 前端通过useChat去消费这个流式消息 在消息开始,结束,失败时分别处理loading和状态切换

在后端将前端消息转发到AI服务器之前,我还做了一层结构清晰, 因为前端传来的消息不光包含role/content, 还会包含一些额外的字段, 例如用户id等, 而ai需要的字段只是role/content, 所以我在转发之前做了一层结构清洗, 只保留模型需要的字段, 避免模型因为接收到过多无关信息而产生理解困难或是输出不稳定的问题.

在模型配置中我接入了一个tool工具调用
当用户问"我今天吃的怎么样/分析今日饮食"等问题时,模型会调用这个工具,工具内部会去数据库查询用户当日的饮食记录,并把这些真实数据返回给模型,模型再结合这些数据生成个性化的回答. 这样就让模型的回答不再是空洞的通用建议,而是基于用户实际情况的智能分析和建议,大大提升了问答的实用性和个性化程度.

并且我做了消息的持久化 在AI回复完成后通过onFinshi再把完整回答写入数据库, 并且前端在回复完毕后重新获取聊天记录确保拿到正确的ID和持久化状态

### 追问

- 为什么要做流式输出？
  - 因为 AI 响应耗时通常比普通接口长，流式返回可以降低等待焦虑，提升交互感知速度
- 为什么不用普通聊天接口一次性返回？
  - 可以做，但体验差；流式输出更像真实对话，也更适合问答类产品
- 工具调用具体是怎么工作的？
  - 模型根据 system prompt 和 tool description 判断是否需要调用工具；如果用户问的是今日饮食分析，就触发 mealInfoAnalysis ，后端查库返回当天 MealItem 数据，模型再基于这些结构化数据生成回答
- 为什么不直接在 prompt 里塞数据库数据？
  - 因为工具调用更动态、更可控，不用每轮都无差别塞大量上下文，成本和逻辑都更合理
- 如果工具返回空数据怎么办？
  - 当前 prompt 已说明：如果工具结果为空，再退回通用建议，不直接胡编用户吃了什么
- 流式输出结束后为什么还要重新请求历史记录？
  - 因为前端流式展示的是当前会话态，重新拉历史记录是为了拿到数据库中的真实记录和 id，保持前端状态与持久化状态一致

基于 Express + Sequelize + MySQL 设计用户、健康档案、饮食记录与聊天记录等核心数据模型，并通过业务校验与查询条件约束保障同用户同日同餐次数据一致性

  - “单纯 findOne + create 在极端并发下理论上会有竞态，所以真正兜底的是数据库联合唯一索引。后续如果再优化，我会把创建逻辑包进事务，或者直接捕获唯一索引冲突做幂等处理。”
- 为什么拆成两张表
  - “MealItem 负责记录单餐明细，Record 更像按天聚合后的汇总视图，两者职责不同。这样一方面保留原始数据，另一方面方便仪表盘直接读取日级汇总。”
