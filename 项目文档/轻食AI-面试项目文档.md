# 轻食AI（基于大模型的健康饮食助手）面试项目文档

## 1. 项目定位（30 秒版本）
- **项目目标**：做一个面向个人用户的健康饮食管理平台，打通“记录饮食 → AI识别/分析 → 个性化建议 → 对话咨询”的闭环。
- **我的角色**：前后端开发负责人，负责前端核心页面、后端 API、数据库模型设计、AI 能力接入与联调。
- **核心价值**：把大模型能力嵌入真实业务流，不只做聊天，而是让 AI 基于用户当日饮食和健康档案给建议。

## 2. 业务闭环（面试可直接口述）
1. 用户注册/登录，拿到 token 并进入主应用。
2. 在首页/分析页录入饮食：可手填，也可上传餐食图片做 AI 识别并估算热量。
3. 饮食数据入库并同步到仪表盘，展示当日/昨日对比和总热量。
4. 用户维护健康档案（基础信息、目标等），系统结合档案生成每日建议。
5. 用户在聊天页提问，模型可调用工具读取当日饮食记录，再输出更个性化回答。

## 3. 技术架构

### 前端
- **技术栈**：React + TypeScript + Vite + React Router + Ant Design + Ant Design X + Zustand + Axios。
- **分层方式**：
  - `page`：页面编排（登录、仪表盘、分析、聊天、个人中心）
  - `components`：复用 UI 与业务组件
  - `api`：按领域拆分接口模块
  - `store`：登录态与建议缓存
  - `utils/http`：请求实例与拦截器

### 后端
- **技术栈**：Express + Sequelize + MySQL + OpenAI SDK + AI SDK。
- **核心模块**：
  - 鉴权与用户管理（登录、注册、个人信息）
  - 饮食记录 CRUD（按用户+日期+餐次）
  - AI 视觉识别（图片识别食物与热量）
  - AI 对话（流式回复 + 工具调用读今日饮食）
  - AI 建议生成（基于健康档案输出 JSON 建议）

### 数据模型
- `User`：账户与基础身份
- `HealthyInfo`：健康档案
- `MealItem`：饮食记录
- `ChatHistory`：对话历史
- **关键约束**：`MealItem` 通过 `(userId, date, mealType)` 联合唯一索引避免同日同餐重复数据。

## 4. 我主导的关键实现（面试亮点）

### 4.1 鉴权闭环统一
- 登录后服务端签发 JWT，前端存入 Zustand + localStorage。
- 请求层通过 Axios 请求拦截器统一注入 token。
- 路由层通过守卫组件拦截未登录访问，避免每个页面重复写校验逻辑。
- **价值**：鉴权逻辑单点维护、接口调用一致性更好。

### 4.2 AI 识图结构化落库
- 用户上传餐食图片后，后端将图片转 base64 送入视觉模型。
- 通过强约束 System Prompt + `response_format`，要求模型只返回 JSON。
- 解析 `items` 列表后，映射为业务字段并入库。
- **价值**：让“看图识别”从演示能力变成可运营的数据能力。

### 4.3 对话式 AI + 业务工具调用
- 聊天接口用流式返回提升交互实时感。
- 模型侧配置工具 `mealInfoAnalysis`，在用户问“今天吃得怎么样”时自动查询当日饮食数据。
- 对前端消息格式做兼容清洗（`content` 与 `parts`），避免上下文字段噪音影响模型理解。
- **价值**：回答不再泛泛而谈，能基于用户真实记录输出建议。

### 4.4 数据一致性与幂等控制
- 写入前通过业务条件查询避免重复创建。
- 数据库层加联合唯一索引做最终兜底。
- 餐次值做标准化映射（大小写与枚举统一）。
- **价值**：减少脏数据、降低线上异常率。

### 4.5 前端状态与性能体验
- 路由懒加载降低初始加载体积。
- 饮食建议按“用户+日期”缓存，避免同日重复调用 AI。
- 聊天消息以 Markdown 渲染，提升可读性与表达能力。

## 5. 面试官高频问题与参考回答

### Q1：你这个项目的技术难点是什么？
**答题框架**：
- 第一是“AI 输出不稳定”，我通过 JSON 强约束、解析兜底与字段清洗提升可用性。
- 第二是“业务一致性”，我用接口前置校验 + DB 联合唯一索引双保险。
- 第三是“对话落地业务”，通过工具调用把聊天和真实饮食数据打通。

### Q2：为什么前端还要做状态持久化？
**答题框架**：
- token 与每日建议都属于“跨刷新仍需保留”的状态。
- 用 Zustand + localStorage 让用户刷新后不用重复登录或重复触发建议生成。
- 结合日期 key 控制缓存粒度，避免旧建议污染新一天。

### Q3：你如何保证 AI 结果可控？
**答题框架**：
- 输入侧：限制上下文字段，避免无关噪音进入模型。
- 输出侧：固定 JSON 模板 + 解析失败兜底数据。
- 业务侧：关键字段做类型转换和默认值保护，防止前端渲染崩溃。

### Q4：如果并发写入同一餐次，会发生什么？
**答题框架**：
- 先在业务层用条件查询拦截大部分重复请求。
- 再由数据库联合唯一索引兜底并返回冲突错误。
- 前端识别冲突提示用户“该餐次已存在，可执行更新”。

### Q5：这个项目还有哪些可优化点？
**答题框架**：
- 把 AI 调用抽象成统一网关，增加限流、重试和熔断。
- 增加接口级监控与埋点（耗时、错误率、命中率）。
- 完善自动化测试：鉴权链路、AI 解析、核心路由集成测试。
- 图片存储可切换对象存储，降低本地磁盘依赖。
