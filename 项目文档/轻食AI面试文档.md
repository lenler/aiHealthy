# 介绍
轻食AI是一个面向个人用户的健康饮食助手，核心目标是解决“饮食记录成本高、建议泛化、难以持续追踪”的问题。我在项目中负责前后端核心链路开发，重点主导前端业务体验与AI交互流程设计。项目打通了“餐食记录（手动/图片）→ AI识别热量 → 结果确认入库 → 仪表盘展示 → 个性化建议/对话咨询”的闭环。

前端采用 React + TypeScript + Ant Design + Zustand，完成登录注册、饮食分析、智能识图、聊天咨询、个人中心等模块，并通过请求层封装与拦截器实现 token 注入、异常统一处理和 API 模块化，提升可维护性与联调效率。在识图链路中，我重点优化了上传与识别体验：设计上传中状态锁、识别草稿确认、失败重试和手动录入降级，确保模型不稳定时核心流程仍可完成。

后端基于 Express + Sequelize + MySQL 提供业务支撑，并通过业务校验与联合唯一约束保障“同用户、同日、同餐次”数据一致性。AI 侧接入视觉识别、建议生成和对话能力，结合工具调用读取当日饮食记录，使回答从通用问答升级为基于用户真实数据的个性化建议。整体上，我在该项目中不仅完成了功能落地，也重点关注了数据可靠性、交互连续性和系统可扩展性。

# 问题:
## **一、核心交互与状态管理**
### 草稿数据的持久化策略：

#### 你是如何存储这些“临时草稿”数据的？如何确保用户确认前的数据不会污染正式的饮食记录状态？

**回答**：Zustand + Persist 我使用了 Zustand 的 persist 中间件将 draftData 存储在 localStorage 中。即使用户刷新页面，识别结果也不会丢失，体验更好。正式数据 ( mealData )：仅在 Analysis 组件内部通过 useState 管理，数据源是后端数据库。草稿数据 ( draftData )：在 Zustand 全局 Store 中管理，数据源是 AI 识别接口。只有当用户 手动点击 “确认添加”按钮时，才会触发数据合并逻辑。在此之前，无论 draftData 如何变化（重新识别、删除草稿），都不会对页面左侧的“当日饮食记录”产生任何影响。

#### 如果用户刷新页面或意外关闭浏览器，如何保证“已识别但未确认”的草稿数据	不丢失？

**回答:**在项目里 AI 识图草稿目前仅存在组件级 state 中，刷新或关闭浏览器会丢失，没有持久化机制；清空草稿依赖用户点击“清空结果”按钮触发,如果要做这个内容可以现在将结果持久化

#### 你会选择 `localStorage`、`sessionStorage` 还是 IndexedDB？为什么？

**回答:** 我会选择localStorage 因为草稿数据本身没有特别大 不包含图像或者大对象 而且需要在退出浏览器后还能持久化 那localStorage是最好的选择

#### 如何在 Zustand 中结合持久化中间件（persist middleware）实现草稿的自动保存与清理？

**回答:** 在Zustand中实现持久化,先创建一个draftStore,并且可以使用persist中间件,然后使用createJSONStorage方法将结构化数据存储到localstrage中,在用户将草稿数据提交入库 或是用户点击删除草稿 或是用户重新让ai识别图片 都会自动将草稿数据清空并且将持久化的数据清空

### 路由切换与状态保持：
#### 当用户在“上传中”或“结果确认页”切换到其他路由（如查看历史记录）再返回时，当前的任务状态（L2oading 或 草稿）是否应该保留？

**回答: **我把草稿数据设计为可恢复的全局状态，路由切换不会丢；上传任务启动后会在后端会将识别后的信息返回给全局状态,所以切换路由当前数据不会丢失

#### 在 React Router 架构下，如何利用 Zustand 的全局特性确保组件卸载后状态不丢失，同时避免内存泄漏？

**回答:** 先将数据提升到 Zustand同时持久化到 localStorage/IndexedDB，组件卸载后仍可恢复；“上传任务状态”可放到全局队列（taskId + status），路由切换后依然能查询到进度；组件卸载时用 useEffect 清理订阅/定时器，避免内存泄漏（例如取消轮询或上传监听）。

### 并发控制,防抖节流与异步流程处理：

#### 如果用户快速点击“确认入库”，或者在网络波动下重复提交，前端如何防止产生脏数据？后端联合唯一索引失效时，前端如何配合？

**回答:**从前端视角我会在“确认入库”按钮上做同步锁（ref/in-flight flag）+ 点击节流，配合请求层去重 生成请求映射表 用请求方式与路径来做唯一key值,来保证一次点击只发一次请求,如果发送多个相同个请求就拦截保护；若后端唯一索引失效或幂等不可用，前端仍用同一个 clientRequestId/业务主键做“本地幂等缓存”，对重复响应只更新 UI 不重复落库，同时在网络波动重试时复用同一 ID 防止生成两条记录，并在 UI 给出“处理中/已提交”的状态提示以阻断重复提交。

#### 虽然 UI 层做了按钮禁用，但在网络极快或用户通过脚本操作的情况下，如何从逻辑层（Request Interceptor 或 Store Action）彻底杜绝重复提交？

**回答:**我会在节流和 UI 禁用之外，在做请求层的锁和去重，避免脚本或极端网络条件下的重复提交。在请求层，使用map数据结果，并通过“请求方法-网址“的形式动态生成key值，判断是否已经存在相同的请求操作 如果存在就拒绝请求并提示用户请勿重复提交 在响应拦截器中获取key值 将map中的数据删除掉 放置下一次请求被拦截

#### ！ 对于图片上传这种大文件操作，是否有做分片上传或进度监控？

**回答:**没做 因为我限制了大文件操作,将图片文件限制在了5mb之下

#### ！ 你是否实现了客户端的超时中断机制（如 `AbortController` 或 `Promise.race`）？

**回答:**现有代码没有实现客户端主动中断或超时控制（无 AbortController/cancel token）。

### 关于“AI识别图片”的有关问题：

#### 请描述一下从**“上传图片” -> “接收 AI 响应” -> “用户确认/修改” -> “最终写入数据库”**这一整个过程中的前端数据结构变化。

**回答:**在用户上传阶段,前端使用antd的组件 将图片封装成FormData对象,后端就会将图片上传到oss然后获取URL,并调用AI对图片镜像识别,最后AI返回原始的JSON数据,后端将数据包装好后发回前端渲染

#### 如果用户在上过图片的程中直接关闭浏览器或刷新页面，你有什么机制来处理这种中断情况吗？

**回答:**

针对用户上传图片时的意外中断和重复提交问题，我设计了一个 三层防护机制 。

第一层是 UI 层的组件级锁定 。我不仅用了 useState 控制上传组件的 disabled 状态，为了防止极短时间内的脚本并发，我还专门引入了 useRef 做同步锁。因为 Ref 的更新不依赖 React 渲染周期，这能百分之百拦截毫秒级的连点。

第二层是 网络层的请求级拦截 。我在 Axios 的拦截器里维护了一个 pendingRequests Map。对于 POST 请求，一旦发现相同 URL 的请求还在处理中，拦截器会直接物理去重，将其 reject 掉。这保证了就算 UI 层的限制失效，后端也只会收到一次真实请求。

第三层是 系统层的页面防护 。我在组件里监听了 window.beforeunload 事件。只要上传状态为 true，如果用户手滑点到了刷新或者关闭浏览器，系统会强制弹窗挽留，防止意外中断。

不过，这个方案目前在 路由级中断 上还有局限。因为状态是绑定在当前页面的组件上的，如果用户点导航切到了其他页面，组件一卸载，虽然请求可能在后台跑完，但回调函数就失效了，用户也看不到结果。

所以，我考虑的 后续优化方案 是：将这个上传任务从‘组件级’提升到‘全局级’。利用 Zustand 维护一个全局的上传任务队列，配合 Axios 的 AbortController 。这样不仅能支持跨页面的后台上传和进度展示，还能在用户真正想取消时，主动掐断网络请求，彻底避免内存泄漏和数据不一致的问题。

#### 在上传图片等待 AI 响应的过程中，如果网络超时或后端无响应，前端如何避免“永久 Loading”的死锁状态？

**回答:**目前靠 Axios 超时 + Upload 内部状态结束上传，但如果后端无响应，UI 只是报错提示，没有显式 “loading 解锁” 状态位。

#### ！ 当 AI 模型识别失败、超时或输出格式不稳定时，你的前端做了什么兜底策略？如何设计“手动录入降级”流程以保证核心业务不中断？

#### 当 AI 识别失败时，你的“重试”逻辑是简单地把原请求再发一遍，还是会根据错误类型做不同的处理？

**回答:**当AI识别失败的时候 我的逻辑是先判断错误类型,如果是网络错误就提醒提醒用户网路连接发生错误,如果是服务器内部出现错误或是AI识别失败 没能识别出热量或是食物种类就会降级会一套固定错误提示数据展示在草稿数据区 提示用户重新识别 或是手动录入

## 二，技术架构与选型决策
### 状态管理选型（Zustand vs Redux/Context）：
#### 为什么在这个项目中选择 Zustand 而不是 Redux Toolkit 或 React Context？

**回答：**就这个项目而言 使用Redux就太浪费了 因为项目体量没有特别大，不需要那么繁重负责的全局状态管理，Zustand简单易用，开始我是考虑到部分页面的全局状态可能用到异步操作，一些异步操作直接就使用async await了 不用像reudx那样用一些中间件实现。Zustand更轻量 而且比原生的React Context更方便性能更好，基本上让我零成本实现了草稿保存 token持久化等功能

#### 在大型应用中，Zustand 如何避免不必要的组件重渲染？

**回答**：在大型应用中，我主要通过 原子化选择器 (Atomic Selectors) 来避免重渲染，确保组件只监听它真正用到的数据。对于需要同时获取多个状态的场景，我会配合 useShallow 进行浅比较，防止因为对象引用变化导致的无效渲染。这比 Context API 的全量更新机制性能要好得多

#### 你是如何组织 Store 结构的？是按功能模块切片（Slice）还是按业务领域？

**回答：**“目前项目规模适中，我是按照业务领域来组织Store结构的 采用 多 Store 独立管理 的策略，比如 authStore 负责用户认证， infoStore 负责饮食数据。这种方式简单直观，避免了巨型 Store 的臃肿。如果未来业务逻辑复杂了，我会引入 Slice Pattern 在单个 Store 内部进行切片管理，保持接口的统一性

### TypeScript 的类型安全实践与前端工程化，模块化：：

如果识别结果确实为空或异常（比如所有字段都是‘未知’），UI 会引导用户‘重新识别’或‘手动录入’，而不是直接报错

### 请求层封装

#### 你提到了“API 模块化”和“请求层封装”。请具体说说你的 Axios 拦截器做了哪些统一处理（Token 注入、全局错误码映射、白名单机制等）？

**回答：**我在Axios拦截器中主要做了两件事 1.**统一的鉴权拦截**, 在请求拦截器中我将发送的请求拦截 并为请求头注入了token，如果我没有从状态中拿到token就直接拦截请求 禁止发送请求 2. **全局防重复提交**,我在请求拦截器中拦截了重复的POST请求 通过创建一个映射表，并将：“请求方法-请求地址“作为键值 然后每次请求之前我都会拦截请求并且将在映射表中检查相同类型的请求是否发送过了 如果已经发送了就拦截本次请求 然后在响应拦截器中拦截响应 检测“请求方法-请求地址”键值 如果检测到存在就删除这个值保证下一次请求能够正常发送

### AI 流式响应的处理：
##### 如果 AI 的建议或对话是流式输出的，前端是如何实现打字机效果的？

如果不使用AISDK或是一些组件库的话 前端可以使用fetchAPI获取ReadableStrem数据，并使用TextDecoder将二进制流解码为文本，然后将些文本追加到消息字符串上。在我的项目中使用了AI SDK和AntdX组件库 在数据层：我使用AI SDK提供的useChat hook（这个hook封装了流式流式请求的一些配置，包括建立链接发送prompt 去读流数据并解析为消息列表 通过onFinish处理生命周期问题）在渲染层我使用了AntdX提供的Bubble组件 用过配置typing属性配置打字机的动画，并是通用XMarkdown组件 处理流式数据的markdown渲染

如果面试官追问细节，可以从以下三个维度展示深度：

1. 流式 Markdown 的解析挑战 ：
   
   - 问题 ：Markdown 渲染器通常需要完整的字符串。如果流只传输了一半的加粗标记 **Bold ，直接渲染会导致格式失效。
   - 解决 ：使用支持 AST（抽象语法树）容错的解析库（如 remark 生态或特制的流式解析器），在解析前自动补全缺失的闭合标签，或者在渲染层做防御性处理。
   
2. 性能优化 (React 渲染) ：
   
   - 问题 ：SSE 频率极高（每秒几十次），如果每次收到数据都触发 React 重渲染（Re-render），会导致页面卡顿。
   - 解决 ：
     - 节流 (Throttle) ：不直接渲染每一帧，而是通过 requestAnimationFrame 或定时器将多次流更新合并为一次 UI 渲染。
     - 非受控组件 ：对于纯展示的文本，可以直接操作 DOM 节点的 textContent 或使用 Ref，避开 React 的 Diff 过程（虽然 Ant Design X 内部可能已经优化了这一点）。
   
3. 自动滚动 (Auto-scroll) ：
   
   - 体验 ：随着文字增加，页面需要自动滚动到底部。
   - 细节 ：需要判断用户是否 手动向上滚动 了。如果用户正在查看历史消息，就应该暂停自动滚动，避免干扰用户阅读；当用户滚回底部时，再恢复自动跟随。
   
   ```tsx
   // 你的代码片段简化理解
   import { useChat } from '@ai-sdk/react';
   import { Bubble, XMarkdown } from '@ant-design/x';
   
   const ChatPage = () => {
     // 1. 数据层：自动处理流
     const { messages, input, handleInputChange, handleSubmit } = useChat();
   
     return (
       <div>
         {messages.map((m) => (
           <Bubble
             key={m.id}
             // 2. 渲染层：处理流式 Markdown 内容
             content={m.content}
             contentRender={(content) => (
               <XMarkdown content={content} />
             )}
             // 3. 交互层：配置打字动画参数
             typing={{ step: 5, interval: 30 }} 
           />
         ))}
       </div>
     );
   };
   ```
   
   

##### 在流式接收过程中，如果用户中途打断或发送新消息，前端如何处理未完成的流？

1. 使用AbortController 这是浏览器原生 API，用于取消尚未完成的 Web 请求。

- 原理 ：在发起 fetch 请求时，传入一个 AbortSignal 信号对象。当调用 controller.abort() 时，浏览器会立即切断该请求的 TCP 连接，读取流的循环（Reader Loop）会收到错误或结束信号，从而停止接收数据。

​	2. 使用useChat等AISDK 内部已经封住那个好这些操作

### 全栈鉴权与数据隔离
#### 作为一个多用户系统，你是如何设计鉴权机制的？如何防止用户 A 查到用户 B 的饮食记录？”

在当前的项目中，我采用了基于 JWT 的无状态鉴权方案，并在数据库层面实现了严格的用户数据隔离。用户登录成功后，后端使用 jsonwebtoken 生成一个设置了过期时间的 Token，其中 Payload 包含了该用户的唯一标识 userId，前端利用 Zustand 和 localStorage 持久化存储 Token。通过配置 Axios 的请求拦截器，在每一次受保护的 API 请求（如获取饮食记录）头中自动注入 Authorization: Bearer   在数据库设计中 我在设计用户数据的时候强关联了一个外键userid 当后端接收不到请求的时候 执行ORM查询 必须将userID作为where语句的约束条件 保证即使用户A登录 系统在数据库层面也只会拿到用户A的数据。

虽然上述方案实现了基本的业务逻辑，但我意识到目前的架构存在一个常见的安全风险，目前后端接口的数据隔离依赖于前端通过 URL 或 Body 传过来的 userId （如 req.params.userId ）。如果恶意用户 A 登录后，手动篡改请求 URL，将自己的 userId 替换成用户 B 的 userId ，系统就会错误地返回用户 B 的数据。

防范用户 A 查到用户 B 的数据，核心原则是 建立一条不可篡改的信任链 。从用户登录获取 JWT，到中间件解析 JWT 提取可信的 userId ，再到 ORM 查询时强制应用这个 userId 进行过滤，最后配合前端的请求拦截器，形成完整的闭环。绝不能单纯依赖前端传递的参数来进行数据权限的参数来作为查询依据（如 URL 参数）传递的身份标识。

**重点**. 局限性与改进：防范“越权漏洞” (IDOR) 话术模板 ：“虽然上述方案实现了基本的业务逻辑，但我意识到目前的架构存在一个常见的安全风险： 水平越权访问（IDOR） 。这是我接下来优化的重点。”

当前风险： 目前，后端接口的数据隔离依赖于前端通过 URL 或 Body 传过来的 userId （如 req.params.userId ）。如果恶意用户 A 登录后，手动篡改请求 URL，将自己的 userId 替换成用户 B 的 userId ，系统就会错误地返回用户 B 的数据。

优化方案（必须在面试中提出）：

- 后端不可信任前端数据 ：永远不要依赖前端传来的 userId 进行数据查询。
- 中间件校验 ：我会实现一个全局的鉴权中间件（Auth Middleware）。在每个受保护的路由前，该中间件会拦截请求，验证 JWT 的合法性，并从中解析出 userId ，然后将其挂载到 req.user 上。
- 数据查询闭环 ：在具体的业务路由（如获取饮食记录）中，直接从 req.user.userId 取值作为数据库查询条件，彻底堵死越权漏洞。

高级架构视野：扩展性与审计 话术模板 ：“为了应对未来系统规模的扩大和更严格的安全合规要求，我认为还需要在架构层面进行补充。”

- 数据层级的行级安全 (Row-Level Security, RLS) ：如果使用的是 PostgreSQL 等支持 RLS 的数据库，可以把隔离逻辑下沉到数据库层面。即使用户（或开发人员的 Bug）在应用层忘了写 where userId = ? ，数据库自身也会拒绝返回非当前用户的数据，提供最后一道防线。
- 审计日志 (Audit Logging) ：对于敏感数据（如健康/饮食记录），记录谁在什么时间访问了什么数据，这不仅有助于事后追踪异常行为，也是医疗/健康类应用合规性的常见要求

**STAR 案例：解决 AI 对话中的“消息丢失”竞态问题**

**情景（S）**

在开发 aiHealthy 的智能问答模块时，我采用了 Vercel AI SDK 配合后端的 SSE (Server-Sent Events) 来实现打字机效果。用户体验要求是：AI 回答完毕后，前端需要立即通过 API 拉取最新的历史记录，以确保数据持久化成功并更新左侧的会话列表。

**任务 (Task)**
我发现了一个偶发的 Bug ：当 AI 回答结束（前端触发 onFinish 回调）并立即请求“获取历史记录列表”接口时，最新的那条 AI 回复经常 凭空消失 。用户刷新页面后，这条记录又出现了。
我的任务是定位这个“幽灵数据”的原因，并保证用户在对话结束后能 100% 看到最新状态，不能有任何延迟感或数据不一致。

**行动 (Action)**
第一步：**定位问题 (Debug & Analysis)**

- 我首先排查了网络时序。通过 Chrome DevTools 发现，AI 的流式响应结束（ Stream Finished ）和前端发起 /api/chat/history 请求几乎是在同一毫秒发生的。
- 深入后端源码 chat.js 分析，我发现后端使用了 onFinish 钩子来异步将 AI 的完整回复写入 MySQL 数据库。
- 根因找到 ：这是一个经典的 分布式竞态条件 (Race Condition) 。
  - 线程 A（流式响应） ：AI 推送完最后一个字符 -> 触发后端 onFinish -> 开始 ChatHistory.create() 写入数据库（IO 操作，耗时约 10-50ms）。
  - 线程 B（前端查询） ：前端收到流结束信号 -> 立即发起 /api/chat/history 查询 -> 数据库查询执行。
  - 结果 ：由于数据库写入是异步的，查询请求往往在写入提交（Commit）之前就执行了，导致查不到最新那条记录。

**第二步：分析方案 (Solution Design) 我构思了三种方案：**

1. 前端延迟（不可靠） ：前端 setTimeout 500ms 再查。
   - 缺点 ：网络波动或数据库抖动时 500ms 不一定够，且让用户白等，体验差。
2. 后端同步阻塞（影响性能） ：后端等数据库写入成功后，再发送流的 [DONE] 信号。
   - 缺点 ：会阻塞 AI 的响应流，导致最后几个字“卡顿”一下，且 Vercel AI SDK 的流式设计本身偏向非阻塞。
3. 乐观更新 (Optimistic UI) + 最终一致性（我的选择） ：
   - 前端不依赖后端的查询结果来展示最新消息，而是直接信任当前的 messages 状态。
   - 仅在后台静默同步数据，确保 ID 等元数据一致。

**第三步：实施解决 (Implementation) ：**
"由于后端 SDK 封装较深，我采用了 **乐观更新 (Optimistic Update) 策略**。在前端 onFinish 触发时，我 不清除 当前的显示列表，也不立即用后端返回的列表覆盖当前屏幕。而是：

1. 保持当前 UI 不变（用户看到的是完整的）。
2. 使用 SWR 或 React Query 的 mutate 方法，在后台静默验证数据。
3. 如果后端数据暂时缺失，UI 依然展示本地缓存，直到下一次轮询或交互对齐数据。" 4. 结果 (Result)
- Bug 消除 ：彻底解决了“消息刚发完就消失”的灵异现象。
- 体验提升 ：用户感觉不到任何加载延迟，对话流畅度提升。
- 代码健壮性 ：通过这次重构，我深刻理解了流式系统中的 端到端时序问题 ，并为此后的即时通讯功能建立了“本地优先”的开发准则。

**面试官可能的追问 (Preparedness)**

- Q: 为什么不直接用 WebSocket？
  - A: 对于 AI 对话这种“一问一答”且主要是下行流量的场景，SSE (Server-Sent Events) 更轻量，且兼容 HTTP/2，不需要处理 WebSocket 的心跳保活和复杂的握手协议。
- Q: 如果写入数据库失败了怎么办？
  - A: 这就是乐观更新的代价。我们需要在前端实现一个“重试/错误提示”机制。如果后端明确返回了写入失败的信号（可以通过流中的 error event），前端会将该消息标记为红色感叹号，允许用户点击重试。
