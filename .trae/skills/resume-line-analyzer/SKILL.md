# Resume Line Analyzer

## Purpose
把简历中的单条项目描述拆成“我到底做了什么、代码证据在哪里、面试时怎么讲、有哪些边界”的完整材料，避免只停留在一句抽象表述。

## When to invoke
- 用户要求“分析简历中的某一句/某一段”
- 用户要求“这句话我具体做了哪些工作”
- 用户要求“这条项目经历怎么讲给面试官听”
- 用户要求“帮我验证这句简历是否和代码一致”
- 用户要求“把这句简历改成可面试/可追问版本”

## Inputs
- 用户给出的简历原句或项目描述
- 当前代码仓库
- 相关的项目文档、面试文档、简历草稿

## Required workflow
1. 先定位与该句描述相关的代码证据
   - 优先检查 store、api、utils、page、components、server routes
   - 必要时补充查看项目文档中的面试材料或优化建议
2. 将一句话拆成 3 到 5 个“具体动作”
   - 做了什么
   - 为什么这么做
   - 落在了哪些模块
   - 带来了什么效果
3. 明确边界
   - 哪些点是当前已实现的
   - 哪些点只是规划或可优化方向
   - 避免把未落地能力写成已完成事实
4. 生成三类输出
   - 代码实现拆解版
   - 面试口述版
   - 简历优化版

## Output format

### 1. 实际做了什么
- 用 3 到 6 条拆解该句背后的真实工作
- 每条都尽量对应到代码或模块

### 2. 面试怎么讲
- 给出一段可直接口述的自然表达
- 强调“问题 -> 方案 -> 效果”

### 3. 边界与追问
- 明确哪些点已实现
- 明确哪些点属于优化方向
- 预判面试官可能追问的问题

### 4. 简历改写
- 输出一条更稳、更真实、可被追问的版本
- 如果原句有夸大成分，要主动收敛措辞

## Evidence rules
- 优先使用真实代码证据，不凭空补全
- 若代码和简历表述不完全一致，要明确指出差距
- 如果某项能力只在文档中提过、代码未落地，要标为“规划中/优化方向”

## Tone rules
- 说人话，避免空泛术语堆砌
- 默认使用中文输出
- 结论先行，再展开细节
- 面试表达要自然，不要写成书面论文

## Good examples
- “基于 Zustand + LocalStorage 管理登录态与建议缓存，减少重复请求并提升刷新后的连续体验”
- “二次封装 Axios 请求层与拦截器，实现统一 token 注入、异常抛出与 POST 请求去重”
- “设计 AI 识图上传与结果确认链路，降低误提交与无效操作”

## Anti-patterns
- 只复述原句，不拆实现
- 不看代码，直接脑补
- 把优化方向说成已上线功能
- 只讲技术名词，不讲业务价值
